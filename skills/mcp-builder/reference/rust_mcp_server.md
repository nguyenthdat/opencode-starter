# Rust MCP Server Implementation Guide

## Overview

This document provides Rust-specific best practices and examples for implementing MCP servers using the official MCP Rust SDK ([`rmcp`](https://github.com/modelcontextprotocol/rust-sdk)). It covers project structure, server setup, tool registration with the `#[tool]` macros, input validation with `serde` + `schemars`, error handling, transports, and complete working examples.

`rmcp` is an async-first SDK built on the [tokio](https://github.com/tokio-rs/tokio) runtime. Tools, prompts, and routers are wired up with procedural macros (`rmcp-macros`), and JSON Schemas are generated from your Rust types with [`schemars`](https://github.com/GREsau/schemars).

**For complete SDK documentation, use WebFetch to load:**
`https://raw.githubusercontent.com/modelcontextprotocol/rust-sdk/main/README.md`

## MCP Rust SDK (`rmcp`)

The official MCP Rust SDK provides:
- `ServiceExt` trait with `.serve(transport)` to start a server or client
- The `#[tool]`, `#[tool_router]`, and `#[tool_handler]` macros that generate `list_tools` / `call_tool` wiring
- The `#[prompt]`, `#[prompt_router]`, and `#[prompt_handler]` macros for prompts
- Automatic `inputSchema` / `outputSchema` generation from your `serde` + `schemars` types
- The `ServerHandler` trait for resources, completions, logging, and custom behavior

**IMPORTANT - Use the macro-based APIs:**
- **DO use**: `#[tool_router]` + `#[tool]` + `#[tool_handler]` for tools; `Parameters<T>` for typed input; `Json<T>` for structured output.
- **DO NOT** hand-implement `list_tools` / `call_tool` on `ServerHandler` unless you have a specialized routing need — the macros provide schema generation and dispatch for you.
- For a **tools-only** server, use `#[tool_router(server_handler)]` to skip writing a separate `impl ServerHandler` block.

### Crate versions

`rmcp` is evolving quickly. Pin a recent version and verify on [crates.io](https://crates.io/crates/rmcp) / [docs.rs](https://docs.rs/rmcp). The examples here target `rmcp` 0.16.x. Some type names changed across versions (e.g. older releases used `Content::text`; current releases use `ContentBlock::text`). If a symbol does not resolve, check the docs.rs page for your pinned version.

## Server Naming Convention

Rust MCP servers (the Cargo package) should follow this naming pattern:
- **Format**: `{service}-mcp-server` (lowercase with hyphens, per Cargo package naming)
- **Examples**: `github-mcp-server`, `jira-mcp-server`, `stripe-mcp-server`

The Rust library target derived from that package uses underscores (`github_mcp_server`). The server identity advertised to clients is set via `Implementation` (typically `Implementation::from_build_env()`, which reads `name`/`version` from `Cargo.toml`).

The name should be:
- General (not tied to specific features)
- Descriptive of the service/API being integrated
- Easy to infer from the task description
- Without version numbers or dates

## Project Structure

Create the following structure for Rust MCP servers:

```
{service}-mcp-server/
├── Cargo.toml
├── README.md
└── src/
    ├── main.rs           # Entry point: tokio runtime + transport selection
    ├── server.rs         # Server struct, #[tool_router], #[tool_handler]
    ├── tools/            # Tool implementations grouped by domain (mod.rs + files)
    │   └── mod.rs
    ├── schemas.rs        # serde + schemars input/output types
    ├── client.rs         # API client and shared HTTP utilities
    ├── error.rs          # Error mapping helpers (-> McpError / CallToolResult)
    └── constants.rs      # Shared constants (API_BASE_URL, CHARACTER_LIMIT, etc.)
```

For small servers, a single `main.rs` is acceptable, but extract shared logic (API client, error mapping, formatting) into modules as the server grows.

## Tool Implementation

### Tool Naming

Use snake_case for tool names (e.g., `search_users`, `create_project`, `get_channel_info`) with clear, action-oriented names. The `#[tool]` macro defaults the name to the function name; set `name = "..."` explicitly to add a service prefix.

**Avoid Naming Conflicts**: Include the service context to prevent overlaps:
- Use `slack_send_message` instead of just `send_message`
- Use `github_create_issue` instead of just `create_issue`
- Use `asana_list_tasks` instead of just `list_tasks`

### Tool Structure

Tools are methods on your server struct, annotated with `#[tool]`, inside an `impl` block annotated with `#[tool_router]`. Requirements:
- Take `&self` so tools can share state (HTTP client, caches, config).
- Accept typed input via `Parameters<T>` where `T: Deserialize + JsonSchema`. The `inputSchema` is generated from `T`.
- Provide `description` (or a `///` doc comment) and `annotations`.
- Return `Result<CallToolResult, McpError>` (text/markdown), `Result<Json<T>, E>` (structured), or a plain `String`.

```rust
use rmcp::{
    ErrorData as McpError,
    handler::server::{router::tool::ToolRouter, wrapper::Parameters},
    model::*,
    tool, tool_router,
};
use schemars::JsonSchema;
use serde::Deserialize;

#[derive(Clone)]
pub struct ExampleServer {
    http: reqwest::Client,
    tool_router: ToolRouter<Self>,
}

// Input model: serde for deserialization, schemars for the JSON Schema.
#[derive(Debug, Deserialize, JsonSchema)]
pub struct UserSearchInput {
    /// Search string to match against names/emails (e.g. "john", "team:marketing").
    #[schemars(length(min = 2, max = 200))]
    pub query: String,

    /// Maximum results to return, between 1-100 (default: 20).
    #[serde(default = "default_limit")]
    #[schemars(range(min = 1, max = 100))]
    pub limit: u32,

    /// Number of results to skip for pagination (default: 0).
    #[serde(default)]
    pub offset: u32,

    /// Output format: 'markdown' for human-readable or 'json' for machine-readable.
    #[serde(default)]
    pub response_format: ResponseFormat,
}

fn default_limit() -> u32 {
    20
}

#[tool_router]
impl ExampleServer {
    pub fn new() -> Self {
        Self {
            http: reqwest::Client::new(),
            tool_router: Self::tool_router(),
        }
    }

    /// Search for users in the Example system by name, email, or team.
    ///
    /// This tool searches across all user profiles in the Example platform,
    /// supporting partial matches. It does NOT create or modify users.
    ///
    /// Args:
    ///   - query (string): Search string to match against names/emails
    ///   - limit (integer): Maximum results to return, 1-100 (default: 20)
    ///   - offset (integer): Number of results to skip for pagination (default: 0)
    ///   - response_format ('markdown' | 'json'): Output format (default: 'markdown')
    ///
    /// Returns (JSON format):
    ///   {
    ///     "total": number,        // Total number of matches found
    ///     "count": number,        // Number of results in this response
    ///     "offset": number,       // Current pagination offset
    ///     "users": [
    ///       { "id": string, "name": string, "email": string,
    ///         "team": string, "active": boolean }
    ///     ],
    ///     "has_more": boolean,    // Whether more results are available
    ///     "next_offset": number   // Offset for next page (if has_more)
    ///   }
    ///
    /// Examples:
    ///   - "Find all marketing team members" -> query="team:marketing"
    ///   - "Search for John's account" -> query="john"
    ///   - Don't use to create a user (use example_create_user instead)
    #[tool(
        name = "example_search_users",
        description = "Search for users in the Example system by name, email, or team. \
                       Read-only; supports partial matches and pagination.",
        annotations(
            title = "Search Example Users",
            read_only_hint = true,
            destructive_hint = false,
            idempotent_hint = true,
            open_world_hint = true
        )
    )]
    async fn example_search_users(
        &self,
        Parameters(params): Parameters<UserSearchInput>,
    ) -> Result<CallToolResult, McpError> {
        // Validate constraints that schemars only *describes* (see note below).
        if params.query.trim().is_empty() {
            return Ok(CallToolResult::error(vec![ContentBlock::text(
                "Error: query cannot be empty. Provide a name, email, or 'team:<name>'.",
            )]));
        }

        let data = self
            .api_get(
                "users/search",
                &[
                    ("q", params.query.clone()),
                    ("limit", params.limit.to_string()),
                    ("offset", params.offset.to_string()),
                ],
            )
            .await?;

        let users = data.get("users").and_then(|u| u.as_array()).cloned().unwrap_or_default();
        let total = data.get("total").and_then(|t| t.as_u64()).unwrap_or(0);

        if users.is_empty() {
            return Ok(CallToolResult::success(vec![ContentBlock::text(format!(
                "No users found matching '{}'",
                params.query
            ))]));
        }

        let text = match params.response_format {
            ResponseFormat::Markdown => format_users_markdown(&params.query, total, &users),
            ResponseFormat::Json => build_users_json(total, params.offset, &users).to_string(),
        };

        Ok(CallToolResult::success(vec![ContentBlock::text(text)]))
    }
}
```

## Input Validation with `serde` + `schemars`

In Rust, two crates split the work that Pydantic/Zod do in one:
- **`serde`** deserializes the incoming JSON into your struct (this is your *type* validation — wrong types are rejected automatically).
- **`schemars`** generates the JSON Schema advertised to clients (this is the *description* of constraints).

**IMPORTANT:** `schemars` constraint attributes (`range`, `length`, `regex`, etc.) are emitted into the JSON Schema but are **not enforced at deserialization time**. Treat them as documentation/hints for the client. For hard runtime guarantees, validate explicitly inside the tool (as shown above) or use a validation crate such as [`garde`](https://crates.io/crates/garde) / [`validator`](https://crates.io/crates/validator).

```rust
use schemars::JsonSchema;
use serde::Deserialize;

#[derive(Debug, Deserialize, JsonSchema)]
pub struct CreateUserInput {
    /// User's full name.
    #[schemars(length(min = 1, max = 100))]
    pub name: String,

    /// User's email address.
    #[schemars(regex(pattern = r"^[\w.+-]+@[\w-]+\.[\w.-]+$"))]
    pub email: String,

    /// User's age in years.
    #[schemars(range(min = 0, max = 150))]
    pub age: u8,

    /// Optional list of tags to apply.
    #[serde(default)]
    #[schemars(length(max = 10))]
    pub tags: Vec<String>,
}
```

Notes:
- Field doc comments (`///`) become the property `description` in the schema — always document fields.
- Use `#[serde(default)]` (or `#[serde(default = "fn")]`) for optional fields. `Option<T>` makes a field nullable/omittable.
- Use `#[serde(deny_unknown_fields)]` on the struct to reject unexpected fields (analogous to Zod's `.strict()` / Pydantic's `extra='forbid'`).

```rust
#[derive(Debug, Deserialize, JsonSchema)]
#[serde(deny_unknown_fields)]
pub struct StrictInput {
    pub id: String,
}
```

## Response Format Options

Support multiple output formats for flexibility. Use an enum that derives `Default`:

```rust
use schemars::JsonSchema;
use serde::Deserialize;

#[derive(Debug, Clone, Copy, Default, Deserialize, JsonSchema)]
#[serde(rename_all = "lowercase")]
pub enum ResponseFormat {
    #[default]
    Markdown,
    Json,
}
```

**Markdown format**:
- Use headers, lists, and formatting for clarity
- Convert timestamps to human-readable format
- Show display names with IDs in parentheses (e.g., "John Doe (U123456)")
- Omit verbose metadata
- Group related information logically

**JSON format**:
- Return complete, structured data suitable for programmatic processing
- Include all available fields and metadata
- Use consistent field names and types

## Pagination Implementation

For tools that list resources, return pagination metadata so agents can fetch more:

```rust
use serde_json::{json, Value};

fn build_users_json(total: u64, offset: u32, users: &[Value]) -> Value {
    let count = users.len() as u64;
    let has_more = total > offset as u64 + count;
    json!({
        "total": total,
        "count": count,
        "offset": offset,
        "users": users,
        "has_more": has_more,
        "next_offset": if has_more { Some(offset as u64 + count) } else { None },
    })
}
```

Always respect a `limit` parameter, default to 20-50 items, and never load all results into memory for large datasets.

## Character Limits and Truncation

Add a `CHARACTER_LIMIT` constant to prevent overwhelming responses:

```rust
// In constants.rs
pub const CHARACTER_LIMIT: usize = 25_000; // Maximum response size in characters

fn enforce_limit(mut response: serde_json::Value, items: &mut Vec<serde_json::Value>) -> String {
    let mut text = response.to_string();
    if text.len() > CHARACTER_LIMIT {
        let keep = (items.len() / 2).max(1);
        items.truncate(keep);
        response["items"] = serde_json::Value::Array(items.clone());
        response["truncated"] = serde_json::Value::Bool(true);
        response["truncation_message"] = serde_json::Value::String(format!(
            "Response truncated to {keep} items. Use 'offset' or add filters to see more."
        ));
        text = response.to_string();
    }
    text
}
```

## Error Handling

`rmcp` distinguishes two kinds of failures, and the difference matters for agents:

| Return | Meaning | Does the agent see your message? |
|--------|---------|----------------------------------|
| `Err(McpError)` | **Protocol-level** error: the request was invalid or could not be routed. | Surfaced as a JSON-RPC error; not a normal tool result. |
| `Ok(CallToolResult::error(...))` | **Tool-level** error: the tool ran but failed (not found, rate limited, validation, etc.). | **Yes** — returned as a tool result with `isError: true`. |

**Rule of thumb:** For failures you want the model to read and recover from (the vast majority — "no rows matched", "rate limited", "invalid input"), return `Ok(CallToolResult::error(...))` with an actionable message. Reserve `Err(McpError)` for genuinely malformed requests or unrecoverable internal faults.

```rust
use rmcp::{model::*, ErrorData as McpError};

/// Map an HTTP/client error into an actionable, agent-visible tool error.
fn api_error_result(status: Option<u16>, context: &str) -> CallToolResult {
    let msg = match status {
        Some(404) => format!("Error: {context} not found. Check the ID is correct."),
        Some(401) => "Error: Invalid API authentication. Check the API key.".to_string(),
        Some(403) => "Error: Permission denied. You don't have access to this resource.".to_string(),
        Some(429) => "Error: Rate limit exceeded. Wait before making more requests.".to_string(),
        Some(s) => format!("Error: API request failed with status {s}."),
        None => "Error: Request failed (timeout or connection error). Please retry.".to_string(),
    };
    CallToolResult::error(vec![ContentBlock::text(msg)])
}

// McpError constructors for protocol-level failures:
//   McpError::invalid_params(message, Option<serde_json::Value>)
//   McpError::internal_error(message, Option<serde_json::Value>)
//   McpError::resource_not_found(message, Option<serde_json::Value>)
```

- Don't expose internal errors (stack traces, raw DB errors) to clients.
- Provide specific, actionable messages with next steps.
- Log security-relevant errors server-side with `tracing` (to stderr — see transports).

## Shared Utilities

Extract common functionality (API client, formatting) into reusable methods/functions instead of duplicating logic across tools:

```rust
use rmcp::{model::*, ErrorData as McpError};
use serde_json::Value;

impl ExampleServer {
    /// Shared GET helper for all tools. Returns parsed JSON or a protocol error
    /// only for unexpected transport faults; HTTP status errors are mapped by callers.
    async fn api_get(&self, path: &str, query: &[(&str, String)]) -> Result<Value, McpError> {
        let url = format!("{}/{}", crate::constants::API_BASE_URL, path);
        let resp = self
            .http
            .get(&url)
            .query(query)
            .header("Accept", "application/json")
            .send()
            .await
            .map_err(|e| McpError::internal_error(format!("request failed: {e}"), None))?;

        let status = resp.status();
        if !status.is_success() {
            // Surface as a protocol error here, or bubble status up to build a
            // CallToolResult::error in the tool. Prefer the latter for agent visibility.
            return Err(McpError::internal_error(
                format!("API returned status {}", status.as_u16()),
                None,
            ));
        }

        resp.json::<Value>()
            .await
            .map_err(|e| McpError::internal_error(format!("invalid JSON: {e}"), None))
    }
}
```

## Async/Await Best Practices

`rmcp` runs on tokio. Tools are `async fn` and should use non-blocking I/O:

```rust
// Good: async, non-blocking HTTP via reqwest
async fn fetch_data(client: &reqwest::Client, id: &str) -> Result<serde_json::Value, reqwest::Error> {
    client
        .get(format!("https://api.example.com/resource/{id}"))
        .send()
        .await?
        .json()
        .await
}
```

- Never call blocking I/O (`std::fs`, blocking `reqwest::blocking`, long CPU loops) directly inside a tool — it stalls the runtime. Use `tokio::fs`, async clients, or `tokio::task::spawn_blocking` for CPU-bound work.
- Share one `reqwest::Client` across tools (it pools connections). Store it in the server struct.
- Set timeouts on the client: `reqwest::Client::builder().timeout(Duration::from_secs(30)).build()`.

## Rust Best Practices

1. **Derive what you need**: input types derive `Deserialize, JsonSchema`; structured output types also derive `Serialize`.
2. **Avoid `unwrap()`/`expect()`** in tool paths — convert errors with `?`, `map_err`, or return `CallToolResult::error`.
3. **Share state via `&self`**: keep the `reqwest::Client`, config, and caches in the server struct; the struct must be `Clone` (wrap mutable state in `Arc<Mutex<…>>` / `Arc<RwLock<…>>`).
4. **Use `tracing`** for logs; never print logs to stdout on stdio transport (stdout is the protocol channel).
5. **Constants** live in `constants.rs` in `SCREAMING_SNAKE_CASE`.
6. **Keep tools focused and atomic**; extract shared formatting/pagination helpers.
7. **Run `cargo clippy`** and fix warnings; format with `cargo fmt`.

## Complete Example

```rust
//! MCP Server for Example Service.
//!
//! Provides tools to interact with the Example API (user search, etc.).
//! Selects transport via the TRANSPORT env var: "stdio" (default) or "http".

use std::env;

use rmcp::{
    ErrorData as McpError, ServerHandler, ServiceExt,
    handler::server::{router::tool::ToolRouter, wrapper::Parameters},
    model::*,
    tool, tool_handler, tool_router,
    transport::stdio,
};
use schemars::JsonSchema;
use serde::Deserialize;
use serde_json::{json, Value};

const API_BASE_URL: &str = "https://api.example.com/v1";
const BIND_ADDRESS: &str = "127.0.0.1:8000";

// ---------------------------------------------------------------------------
// Input types
// ---------------------------------------------------------------------------

#[derive(Debug, Clone, Copy, Default, Deserialize, JsonSchema)]
#[serde(rename_all = "lowercase")]
pub enum ResponseFormat {
    #[default]
    Markdown,
    Json,
}

#[derive(Debug, Deserialize, JsonSchema)]
#[serde(deny_unknown_fields)]
pub struct UserSearchInput {
    /// Search string to match against names/emails (e.g. "john", "team:marketing").
    #[schemars(length(min = 2, max = 200))]
    pub query: String,
    /// Maximum results to return, 1-100 (default: 20).
    #[serde(default = "default_limit")]
    #[schemars(range(min = 1, max = 100))]
    pub limit: u32,
    /// Number of results to skip for pagination (default: 0).
    #[serde(default)]
    pub offset: u32,
    /// Output format: 'markdown' or 'json' (default: 'markdown').
    #[serde(default)]
    pub response_format: ResponseFormat,
}

fn default_limit() -> u32 {
    20
}

// ---------------------------------------------------------------------------
// Server
// ---------------------------------------------------------------------------

#[derive(Clone)]
pub struct ExampleServer {
    http: reqwest::Client,
    api_key: String,
    tool_router: ToolRouter<Self>,
}

#[tool_router]
impl ExampleServer {
    pub fn new(api_key: String) -> Self {
        let http = reqwest::Client::builder()
            .timeout(std::time::Duration::from_secs(30))
            .build()
            .expect("failed to build HTTP client");
        Self { http, api_key, tool_router: Self::tool_router() }
    }

    async fn api_get(&self, path: &str, query: &[(&str, String)]) -> Result<(u16, Value), McpError> {
        let resp = self
            .http
            .get(format!("{API_BASE_URL}/{path}"))
            .query(query)
            .bearer_auth(&self.api_key)
            .header("Accept", "application/json")
            .send()
            .await
            .map_err(|e| McpError::internal_error(format!("request failed: {e}"), None))?;
        let status = resp.status().as_u16();
        let body = resp.json::<Value>().await.unwrap_or_else(|_| json!({}));
        Ok((status, body))
    }

    /// Search for users in the Example system by name, email, or team.
    ///
    /// Read-only. Supports partial matches and pagination. Returns either a
    /// markdown summary or a JSON object with total/count/offset/users/has_more.
    #[tool(
        name = "example_search_users",
        description = "Search Example users by name, email, or team. Read-only; supports pagination.",
        annotations(
            title = "Search Example Users",
            read_only_hint = true,
            destructive_hint = false,
            idempotent_hint = true,
            open_world_hint = true
        )
    )]
    async fn example_search_users(
        &self,
        Parameters(params): Parameters<UserSearchInput>,
    ) -> Result<CallToolResult, McpError> {
        if params.query.trim().is_empty() {
            return Ok(CallToolResult::error(vec![ContentBlock::text(
                "Error: query cannot be empty. Provide a name, email, or 'team:<name>'.",
            )]));
        }

        let (status, data) = self
            .api_get(
                "users/search",
                &[
                    ("q", params.query.clone()),
                    ("limit", params.limit.to_string()),
                    ("offset", params.offset.to_string()),
                ],
            )
            .await?;

        if status != 200 {
            let msg = match status {
                401 => "Error: Invalid API authentication. Check EXAMPLE_API_KEY.".to_string(),
                403 => "Error: Permission denied for user search.".to_string(),
                429 => "Error: Rate limit exceeded. Wait before retrying.".to_string(),
                s => format!("Error: API request failed with status {s}."),
            };
            return Ok(CallToolResult::error(vec![ContentBlock::text(msg)]));
        }

        let users = data.get("users").and_then(Value::as_array).cloned().unwrap_or_default();
        let total = data.get("total").and_then(Value::as_u64).unwrap_or(0);

        if users.is_empty() {
            return Ok(CallToolResult::success(vec![ContentBlock::text(format!(
                "No users found matching '{}'",
                params.query
            ))]));
        }

        let text = match params.response_format {
            ResponseFormat::Markdown => {
                let mut lines = vec![
                    format!("# User Search Results: '{}'", params.query),
                    String::new(),
                    format!("Found {total} users (showing {})", users.len()),
                    String::new(),
                ];
                for u in &users {
                    let name = u.get("name").and_then(Value::as_str).unwrap_or("?");
                    let id = u.get("id").and_then(Value::as_str).unwrap_or("?");
                    let email = u.get("email").and_then(Value::as_str).unwrap_or("?");
                    lines.push(format!("## {name} ({id})"));
                    lines.push(format!("- **Email**: {email}"));
                    if let Some(team) = u.get("team").and_then(Value::as_str) {
                        lines.push(format!("- **Team**: {team}"));
                    }
                    lines.push(String::new());
                }
                lines.join("\n")
            }
            ResponseFormat::Json => {
                let count = users.len() as u64;
                let has_more = total > params.offset as u64 + count;
                json!({
                    "total": total,
                    "count": count,
                    "offset": params.offset,
                    "users": users,
                    "has_more": has_more,
                    "next_offset": if has_more { Some(params.offset as u64 + count) } else { None },
                })
                .to_string()
            }
        };

        Ok(CallToolResult::success(vec![ContentBlock::text(text)]))
    }
}

#[tool_handler]
impl ServerHandler for ExampleServer {
    fn get_info(&self) -> ServerInfo {
        ServerInfo::new(ServerCapabilities::builder().enable_tools().build())
            .with_server_info(Implementation::from_build_env())
            .with_instructions(
                "Tools to interact with the Example API. Tool: example_search_users.",
            )
    }
}

// ---------------------------------------------------------------------------
// Entry point: transport selection
// ---------------------------------------------------------------------------

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    // Log to stderr — stdout is reserved for the stdio protocol channel.
    tracing_subscriber::fmt()
        .with_writer(std::io::stderr)
        .with_env_filter(
            tracing_subscriber::EnvFilter::try_from_default_env()
                .unwrap_or_else(|_| "info".into()),
        )
        .init();

    let api_key = env::var("EXAMPLE_API_KEY")
        .map_err(|_| anyhow::anyhow!("EXAMPLE_API_KEY environment variable is required"))?;

    match env::var("TRANSPORT").as_deref() {
        Ok("http") => run_http(api_key).await,
        _ => run_stdio(api_key).await,
    }
}

/// stdio transport (local integrations).
async fn run_stdio(api_key: String) -> anyhow::Result<()> {
    let service = ExampleServer::new(api_key).serve(stdio()).await?;
    service.waiting().await?;
    Ok(())
}

/// Streamable HTTP transport (remote servers, multiple clients).
async fn run_http(api_key: String) -> anyhow::Result<()> {
    use rmcp::transport::streamable_http_server::{
        StreamableHttpServerConfig, StreamableHttpService, session::local::LocalSessionManager,
    };

    let ct = tokio_util::sync::CancellationToken::new();

    // A new service instance is created per session by the factory closure.
    let service = StreamableHttpService::new(
        move || Ok(ExampleServer::new(api_key.clone())),
        LocalSessionManager::default().into(),
        StreamableHttpServerConfig::default().with_cancellation_token(ct.child_token()),
    );

    let router = axum::Router::new().nest_service("/mcp", service);
    let listener = tokio::net::TcpListener::bind(BIND_ADDRESS).await?;
    tracing::info!("MCP server listening on http://{BIND_ADDRESS}/mcp");

    axum::serve(listener, router)
        .with_graceful_shutdown(async move {
            let _ = tokio::signal::ctrl_c().await;
            ct.cancel();
        })
        .await?;
    Ok(())
}
```

## Advanced MCP Features

### Structured Output with `Json<T>`

For machine-readable structured data, return `Json<T>`. `rmcp` auto-generates the tool's `outputSchema` from `T` and serializes the value into the result's `structuredContent`:

```rust
use rmcp::{Json, handler::server::wrapper::Parameters, model::*, tool, ErrorData as McpError};
use schemars::JsonSchema;
use serde::{Deserialize, Serialize};

#[derive(Debug, Serialize, Deserialize, JsonSchema)]
pub struct WeatherResponse {
    pub temperature: f64,
    pub description: String,
    pub humidity: u8,
}

#[derive(Debug, Deserialize, JsonSchema)]
pub struct WeatherRequest {
    pub city: String,
}

// inside #[tool_router] impl:
#[tool(name = "get_weather", description = "Get current weather for a city")]
async fn get_weather(
    &self,
    Parameters(req): Parameters<WeatherRequest>,
) -> Result<Json<WeatherResponse>, McpError> {
    let weather = WeatherResponse {
        temperature: 22.5,
        description: format!("Clear skies over {}", req.city),
        humidity: 65,
    };
    Ok(Json(weather))
}
```

Prefer `Json<T>` when the consumer is programmatic; prefer `CallToolResult` with markdown text when optimizing for agent readability. You can also return both unstructured text and structured content by constructing a `CallToolResult` directly.

### Resource Registration

Expose data as resources by implementing `list_resources`, `read_resource`, and optionally `list_resource_templates` on `ServerHandler`, and enabling the capability in `get_info`:

```rust
use rmcp::{ErrorData as McpError, RoleServer, ServerHandler, model::*, service::RequestContext};
use serde_json::json;

impl ServerHandler for ExampleServer {
    fn get_info(&self) -> ServerInfo {
        ServerInfo::new(
            ServerCapabilities::builder().enable_tools().enable_resources().build(),
        )
    }

    async fn list_resources(
        &self,
        _request: Option<PaginatedRequestParams>,
        _context: RequestContext<RoleServer>,
    ) -> Result<ListResourcesResult, McpError> {
        Ok(ListResourcesResult {
            resources: vec![Resource::new("memo://insights", "insights")],
            next_cursor: None,
            meta: None,
        })
    }

    async fn read_resource(
        &self,
        request: ReadResourceRequestParams,
        _context: RequestContext<RoleServer>,
    ) -> Result<ReadResourceResult, McpError> {
        match request.uri.as_str() {
            "memo://insights" => Ok(ReadResourceResult::new(vec![ResourceContents::text(
                "Analysis results...",
                request.uri,
            )])),
            _ => Err(McpError::resource_not_found(
                "resource_not_found",
                Some(json!({ "uri": request.uri })),
            )),
        }
    }
}
```

> Note: if you implement `ServerHandler` by hand for resources, either keep using `#[tool_handler]` on a separate `impl` block, or call the generated tool dispatch from your manual `call_tool`. The simplest pattern is one `#[tool_handler] impl ServerHandler` block that also contains the resource methods.

**When to use Resources vs Tools:**
- **Resources**: data access via simple URI-based parameters; relatively static/template-based.
- **Tools**: operations with validation, side effects, or complex workflows.

### Transport Options

`rmcp` supports two main transports:

#### stdio (local integrations)

```rust
use rmcp::{ServiceExt, transport::stdio};

let service = ExampleServer::new(api_key).serve(stdio()).await?;
service.waiting().await?;
```

Requires the `transport-io` feature. **Do not log to stdout** — use stderr (`tracing_subscriber::fmt().with_writer(std::io::stderr)`).

#### Streamable HTTP (remote servers, multiple clients)

```rust
use rmcp::transport::streamable_http_server::{
    StreamableHttpServerConfig, StreamableHttpService, session::local::LocalSessionManager,
};

let service = StreamableHttpService::new(
    move || Ok(ExampleServer::new(api_key.clone())),
    LocalSessionManager::default().into(),
    StreamableHttpServerConfig::default(),
);
let router = axum::Router::new().nest_service("/mcp", service);
axum::serve(tokio::net::TcpListener::bind("127.0.0.1:8000").await?, router).await?;
```

Requires the `transport-streamable-http-server` feature plus `axum`. For local deployments, keep DNS-rebinding protection on: bind to `127.0.0.1`, and use `StreamableHttpServerConfig`'s `with_allowed_hosts` / `with_allowed_origins` to restrict `Host`/`Origin` (don't disable them for public deployments).

**Transport selection:**
- **stdio**: Command-line tools, local development, subprocess integration.
- **Streamable HTTP**: Web services, remote access, multiple clients.

### Notifications

When tools/resources/prompts change, notify the client via the peer in `RequestContext`:

```rust
context.peer.notify_tool_list_changed().await?;
context.peer.notify_resource_list_changed().await?;
```

For long-running tools, report progress:

```rust
context
    .peer
    .notify_progress(
        ProgressNotificationParam::new(ProgressToken(NumberOrString::Number(i)), i as f64)
            .with_total(total as f64)
            .with_message(format!("Processing {i}/{total}")),
    )
    .await?;
```

Use notifications sparingly — only when server state genuinely changes.

## Code Best Practices

### Code Composability and Reusability

Your implementation MUST prioritize composability and code reuse:

1. **Extract Common Functionality**:
   - Create reusable helper methods on the server struct (e.g. `api_get`, `api_post`).
   - Centralize error mapping in one place (`error.rs`).
   - Extract shared markdown/JSON formatting and field selection.
   - Extract pagination metadata construction into a helper.

2. **Avoid Duplication**:
   - NEVER copy-paste similar code between tools.
   - If you write similar logic twice, extract it into a function.
   - Authentication/authorization belongs in the shared client, not each tool.

## Quality Checklist

Before finalizing your Rust MCP server implementation, ensure:

### Strategic Design
- [ ] Tools enable complete workflows, not just API endpoint wrappers
- [ ] Tool names reflect natural task subdivisions
- [ ] Response formats optimize for agent context efficiency
- [ ] Human-readable identifiers used where appropriate
- [ ] Error messages guide agents toward correct usage

### Implementation Quality
- [ ] FOCUSED IMPLEMENTATION: Most important and valuable tools implemented
- [ ] All tools defined with `#[tool]` inside a `#[tool_router]` impl block
- [ ] `#[tool_handler]` implemented on `impl ServerHandler` (or `#[tool_router(server_handler)]` for tools-only)
- [ ] All tools include `description` (or doc comment) and `annotations`
- [ ] Annotations correctly set (read_only_hint, destructive_hint, idempotent_hint, open_world_hint)
- [ ] All tool inputs use `Parameters<T>` with `Deserialize` + `JsonSchema` derives
- [ ] Input field doc comments provide schema descriptions; constraints added via `#[schemars(...)]`
- [ ] Runtime validation performed for constraints that must be enforced (schemars is descriptive only)
- [ ] Tool docs include return-value schema and usage examples

### Rust Quality
- [ ] No `unwrap()`/`expect()` on fallible paths in tools; errors handled with `?`/`map_err`/`CallToolResult::error`
- [ ] Server struct is `Clone`; shared mutable state wrapped in `Arc<Mutex<…>>`/`Arc<RwLock<…>>`
- [ ] A single shared `reqwest::Client` (with timeout) stored in the server struct
- [ ] `Err(McpError)` vs `Ok(CallToolResult::error(...))` chosen deliberately (agent-visible errors use the latter)
- [ ] All I/O is async; no blocking calls inside async tools
- [ ] `cargo clippy` is clean; code formatted with `cargo fmt`

### Advanced Features (where applicable)
- [ ] `Json<T>` used for structured output where consumers are programmatic
- [ ] Resources implemented (`list_resources`/`read_resource`) and capability enabled
- [ ] Appropriate transport configured (stdio or streamable HTTP)
- [ ] Notifications/progress used for dynamic capabilities or long operations

### Project Configuration

- [ ] Package name follows format: `{service}-mcp-server`
- [ ] Server identity set via `Implementation` (e.g. `from_build_env()`)
- [ ] stdio servers log to stderr, never stdout

### Code Quality

- [ ] Large responses check `CHARACTER_LIMIT` and truncate with a clear message
- [ ] Filtering options provided for potentially large result sets
- [ ] Network operations set timeouts and handle connection errors gracefully
- [ ] Common functionality extracted into reusable helpers
