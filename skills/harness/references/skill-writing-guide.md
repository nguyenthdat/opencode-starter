## 1. Description-writing patterns

The description is a skill's only trigger mechanism. An agent decides whether to use a skill based purely on the name + description shown in its list of available skills.

### Understanding the trigger mechanism

An agent tends not to reach for a skill when a task looks simple enough to handle with its default tools. A request as plain as "read this PDF for me" might not trigger a skill even with a perfect description. The more complex, multi-step, and specialized the task, the more likely a skill is to trigger.

### Writing principles

1. State both **what the skill does** and **the concrete situations that should trigger it**.
2. Spell out the boundary conditions that distinguish "trigger this" from "looks similar, but don't."
3. Write a little "pushy" — this compensates for an agent's tendency to judge triggering conservatively.

### Good examples

```yaml
description: "Reads PDF files, extracts text/tables, merges, splits, rotates,
  watermarks, encrypts/decrypts, and OCRs — handles every PDF operation. Use
  this skill whenever a .pdf file is mentioned or a PDF deliverable is
  requested. Especially useful beyond a simple 'read this PDF' — for any
  conversion, editing, or analysis task."
```

```yaml
description: "Handles every spreadsheet operation on Excel/CSV/TSV files —
  adding columns, formula calculation, formatting, charts, data cleaning. Use
  this skill whenever the user mentions a spreadsheet file, even casually
  ('the xlsx in my downloads folder')."
```

### Bad examples

- `"A skill for processing data"` — too vague; unclear what file type or operation
- `"PDF-related work"` — no concrete actions listed, no trigger situations described

## 2. Body-writing style

### Why-first principle

An agent that understands the reasoning behind a rule judges edge cases correctly too. Explaining context works better than issuing forceful directives.

**Bad:**
```markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.
```

**Good:**
```markdown
Use pdfplumber for table extraction. PyPDF2 is specialized for text
extraction and doesn't preserve a table's row/column structure, whereas
pdfplumber recognizes cell boundaries and returns structured data.
```

### Generalization principle

When feedback or test results surface a problem, generalize the fix at the level of the underlying principle rather than patching narrowly for one example.

**Overfit fix:**
```markdown
If there's a "Q4 revenue" column, convert it to a number.
```

**Generalized fix:**
```markdown
If a column name contains a keyword implying a numeric quantity — "revenue",
"amount", "count", etc. — convert that column to a numeric type. If the
conversion fails, keep the original value.
```

### Imperative tone

Use direct, instructional phrasing ("do this," "use that") rather than descriptive or conditional phrasing ("you could," "you might"). A skill is a set of instructions.

### Context budget

The context window is a shared resource. Ask, of every sentence: does it earn its token cost?
- "Does the agent already know this?" → cut it.
- "Would the agent get this wrong without this line?" → keep it.
- "Would one concrete example beat a long explanation?" → replace with the example.

## 3. Output-format definition patterns

Use this when the shape of the deliverable matters:

```markdown
## Report structure
Follow this template exactly:

# [Title]
## Summary
## Key findings
## Recommendations
```

Keep format definitions concise; a real example is more effective than more prose.

## 4. Example-writing patterns

Examples are more effective than long explanations:

```markdown
## Commit message format

**Example 1:**
Input: added JWT-token-based user authentication
Output: feat(auth): implement JWT-based authentication

**Example 2:**
Input: fixed a bug where the password-visibility toggle didn't work on the login page
Output: fix(login): fix password-visibility toggle button
```

## 5. Progressive disclosure patterns

### Pattern 1: split by domain

```
bigquery-skill/
├── SKILL.md (overview + domain-selection guide)
└── references/
    ├── finance.md (revenue, billing metrics)
    ├── sales.md (opportunities, pipeline)
    └── product.md (API usage, features)
```

If the user asks about revenue, only `finance.md` gets loaded.

### Pattern 2: conditional detail

```markdown
# DOCX processing

## Creating a document
Generate a new document with docx-js. → See [DOCX-JS.md](references/docx-js.md).

## Editing a document
Simple edits: modify the XML directly.
**If tracked changes are needed**: see [REDLINING.md](references/redlining.md)
```

### Pattern 3: structuring large reference files

Any reference file over 300 lines gets a table of contents at the top:

```markdown
# API Reference

## Table of contents
1. [Authentication](#authentication)
2. [Endpoint list](#endpoint-list)
3. [Error codes](#error-codes)
4. [Rate limits](#rate-limits)

---

## Authentication
...
```

## 6. Criteria for bundling scripts

Observe transcripts from test runs. Bundle when you see these signals:

| Signal | Action |
|------|------|
| The same helper script gets written in 3 out of 3 test runs | Bundle it in `scripts/` |
| The same pip install/npm install runs every time | Specify the dependency-install step in the skill |
| The same multi-step approach repeats every time | Document it as a standard procedure in the skill body |
| The same error keeps happening, followed by the same workaround | Document the known issue and its fix in the skill |

Any bundled script must be execution-tested.

## 7. Data schema standards

Use a standard schema for consistency across data exchanged between skills. These schemas apply to testing/evaluating skills generated by this harness.

### eval_metadata.json

Metadata for each test case:

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "the user's task prompt",
  "assertions": [
    "the output includes X",
    "a file was created in Y format"
  ]
}
```

### grading.json

Assertion-based grading results:

```json
{
  "expectations": [
    {
      "text": "the output includes 'Seoul'",
      "passed": true,
      "evidence": "step 3 confirms 'extracted Seoul-region data'"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}
```

**Field-name note:** use exactly `text`, `passed`, `evidence` (not variants like `name`/`met`/`details`).

### timing.json

Execution time / token measurement:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

Capture `total_tokens` and `duration_ms` **immediately** from the subagent's completion notification. This data is only accessible at that moment and can't be recovered afterward.

## 8. What not to include in a skill

- Supplementary docs like README.md, CHANGELOG.md, INSTALLATION_GUIDE.md
- Meta-information about the skill's own creation (test results, iteration history)
- User-facing documentation (a skill is a set of instructions for an AI agent, not a user manual)
- General knowledge the agent already has

## 9. Skill reuse design

Before creating a new skill, check it against existing ones. Building harnesses repeatedly tends to accumulate skills with overlapping functionality under different names.

| Situation | Action |
|------|------|
| An existing skill fully covers the new functionality | Don't create a new one — wire the existing skill to the agent |
| An existing skill partially covers it and can be generalized | Generalize the existing skill to extend it |
| The overlap is intentional domain specialization | Proceed with a new skill — keep them separate |
| The functionality is entirely different | Proceed with a new skill |

**Principle:** the more a skill focuses on a single responsibility, the more reusable it is and the less duplication accumulates. If a skill covers two or more responsibilities, check first whether it can be split.

### How far to generalize

Generalization can go on indefinitely, so stop at the **intended scope of responsibility**. Keep intentional domain specialization; remove only accidental coupling.

Example: a "fintech risk-assessment PDF" skill

| Step | Result |
|------|------|
| Remove the fintech coupling | "Assessment-result PDF" — if the intended responsibility is assessment reports, stop here |
| Remove the assessment coupling | "PDF formatting" — if this already exists, reuse it rather than creating a separate skill |

If the intended scope really is "fintech risk assessment," don't generalize past that — keep it as a separate skill.

Extending an existing skill can change behavior for any agent that depends on it. Check dependencies before extending, and update the description to reflect the broadened scope.
