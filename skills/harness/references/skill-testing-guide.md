## 1. Testing framework overview

Verifying skill quality combines **qualitative** and **quantitative** evaluation.

| Evaluation type | Method | Good for |
|----------|------|-----------|
| **Qualitative** | User reviews the output directly | Subjective quality — prose style, design, creative work |
| **Quantitative** | Automated assertion-based grading | Objectively verifiable output — file creation, data extraction, code generation |

Core loop: **write → run tests → evaluate → improve → retest**.

## 2. Writing test prompts

### Principle

Test prompts must be **concrete, natural sentences a real user would actually type**. Abstract or artificial prompts have little test value.

### Bad examples

```
"Process a PDF"
"Extract the data"
"Generate a chart"
```

### Good examples

```
"In 'Q4_Revenue_Final_v2.xlsx' in my downloads folder, add a profit-margin
(%) column using column C (revenue) and column D (cost). Then sort
descending by profit margin."
```

```
"Extract the table on page 3 of this PDF and convert it to CSV. The table
header spans two rows — the first row is the category, the second row is
the actual column name."
```

### Prompt diversity

- Mix **formal / casual** tone.
- Mix **explicit / implicit** intent (stating the file format directly vs. requiring it to be inferred from context).
- Mix **simple / complex** tasks.
- Include some with abbreviations, typos, or casual phrasing.

### Coverage

Start with 2–3 prompts, designed to cover:
- one core use case
- one edge case
- (optional) one compound/multi-step task

## 3. Execution testing: with-skill vs. baseline

### 3-1. Comparison structure

For each test prompt, dispatch two subagents **at the same time**:

**With-skill run:**
```
Prompt: "{test prompt}"
Skill path: {skill path}
Output path: _workspace/iteration-N/eval-{id}/with_skill/outputs/
```

**Baseline run:**
```
Prompt: "{test prompt}"  (identical)
Skill: none
Output path: _workspace/iteration-N/eval-{id}/without_skill/outputs/
```

### 3-2. Choosing a baseline

| Situation | Baseline |
|------|----------|
| Creating a new skill | Same prompt, run without the skill |
| Improving an existing skill | The pre-edit version of the skill (keep a snapshot) |

### 3-3. Capturing timing data

Capture `total_tokens` and `duration_ms` from the subagent's completion notification **immediately**. This data is only accessible at that moment and can't be recovered afterward.

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

## 4. Quantitative evaluation: assertion-based grading

### 4-1. Writing assertions

For objectively verifiable output, define assertions for automated grading.

**Good assertions:**
- Objectively true/false
- Named descriptively enough that the result alone tells you what's being checked
- Verify the skill's core value proposition

**Bad assertions:**
- Always pass regardless of whether the skill was used (e.g. "output exists")
- Require subjective judgment (e.g. "well written")

### 4-2. Programmatic verification

Write assertions as scripts wherever they're code-checkable. It's faster and more reliable than eyeballing, and reusable across iterations.

### 4-3. Watch for non-discriminating assertions

An assertion that "passes 100% in both configurations" doesn't measure the skill's differential value. When you find one, remove it or replace it with a more demanding assertion.

### 4-4. Grading result schema

```json
{
  "expectations": [
    {
      "text": "profit-margin column was added",
      "passed": true,
      "evidence": "confirmed column E is 'profit_margin_pct'"
    },
    {
      "text": "sorted descending by profit margin",
      "passed": false,
      "evidence": "original order was kept, no sort applied"
    }
  ],
  "summary": {
    "passed": 1,
    "failed": 1,
    "total": 2,
    "pass_rate": 0.50
  }
}
```

## 5. Using specialized agents

Quality improves when specialized-role agents assist with testing/evaluation.

### 5-1. Grader

Performs assertion-based grading and cross-verifies checkable claims extracted from the output.

**Role:**
- Judges pass/fail per assertion, with supporting evidence
- Extracts factual claims from the output and verifies them
- Feeds back on the eval's own quality (flags assertions that are too easy or too vague)

### 5-2. Comparator (blind comparison)

Anonymizes two outputs as A/B and judges quality without knowing which one used the skill.

**When to use:** when you need a rigorous answer to "is the new version actually better?" Ordinary iteration can usually skip this.

**Judging criteria:**
- Content: accuracy, completeness
- Structure: organization, formatting, usability
- Overall score

### 5-3. Analyzer

Analyzes statistical patterns across benchmark data:
- Non-discriminating assertions (passes in both configurations → no discriminating power)
- High-variance evals (results vary a lot run to run → unstable)
- Time/token trade-offs (the skill raises quality but also raises cost)

## 6. Iterative improvement loop

### 6-1. Collecting feedback

Show the user the output and ask for feedback. Interpret empty feedback as "no issues."

### 6-2. Improvement principles

1. **Generalize the feedback** — a narrow fix that only matches the test example is overfitting. Fix at the level of the underlying principle.
2. **Cut anything that isn't earning its weight** — read the transcript, and if the skill is pushing the agent toward unproductive work, remove that part.
3. **Explain why** — even when the user's feedback is terse, understand why it matters and reflect that understanding in the skill.
4. **Bundle repeated work** — if the same helper script appears across every test run, pre-bundle it in `scripts/`.

### 6-3. Iteration procedure

```
1. Edit the skill
2. Re-run every test case in a new iteration-N+1/ directory
3. Present the results to the user (compared against the previous iteration)
4. Collect feedback
5. Edit again → repeat
```

**Stop when:**
- The user is satisfied.
- Feedback comes back empty (nothing wrong with any output).
- Improvements are no longer meaningful.

### 6-4. Draft → re-review pattern

When editing a skill, write a draft, then **re-read it with fresh eyes** and refine. Don't try to write it perfectly in one pass — go through a draft-then-review cycle.

## 7. Description trigger verification

### 7-1. Writing trigger eval queries

Write 20 eval queries — 10 should-trigger + 10 should-NOT-trigger.

**Query quality criteria:**
- Concrete, natural sentences a real user would type
- Include specific details — file paths, personal context, column names, company names
- Mix length, tone, and format
- Focus on **edge cases**, not obvious cases

**Should-trigger queries (8–10):**
- The same intent phrased differently (formal/casual)
- Cases where the skill/file type isn't named explicitly but is clearly needed
- Uncommon-but-valid use cases
- Cases competing with another skill, where this skill should still win

**Should-NOT-trigger queries (8–10):**
- **Near-misses are the point** — queries with similar keywords where a different tool/skill is actually the right fit
- Obviously unrelated queries ("write a Fibonacci function") have no test value
- Adjacent domains, ambiguous phrasing, overlapping keywords but different intended context

### 7-2. Checking for conflicts with existing skills

Confirm the new skill's description doesn't overlap with an existing skill's trigger territory:

1. Collect the descriptions of existing skills.
2. Confirm the new skill's should-trigger queries don't misfire an existing skill.
3. If a conflict surfaces, sharpen the boundary conditions in the description.

### 7-3. Automated optimization (optional, advanced)

When description optimization is warranted:

1. Split the 20 eval queries into Train (60%) / Test (40%).
2. Measure trigger accuracy with the current description.
3. Analyze failure cases and generate an improved description.
4. Pick the best description based on the Test set (not the Train set — avoid overfitting).
5. Repeat up to 5 rounds.

> Run this as an automated script driving `opencode run` non-interactively. Token cost is high, so run it late — once the skill is already reasonably stable.

## 8. Workspace structure

A directory layout for systematically managing test/evaluation results:

```
{skill-name}-workspace/
├── iteration-1/
│   ├── eval-descriptive-name-1/
│   │   ├── eval_metadata.json
│   │   ├── with_skill/
│   │   │   ├── outputs/
│   │   │   ├── timing.json
│   │   │   └── grading.json
│   │   └── without_skill/
│   │       ├── outputs/
│   │       ├── timing.json
│   │       └── grading.json
│   ├── eval-descriptive-name-2/
│   │   └── ...
│   └── benchmark.json
├── iteration-2/
│   └── ...
└── evals/
    └── evals.json
```

**Rules:**
- Use **descriptive names** for eval directories, not numbers (e.g. `eval-multi-page-table-extraction`).
- Preserve each iteration in its own directory (never overwrite a previous iteration).
- Don't delete `_workspace/` — it's for post-hoc verification and audit.
